# 工程化说明

这个项目刻意保持轻量：它本身就是一个 SillyTavern 扩展目录，没有构建步骤、后端服务或额外运行时依赖。

## 验证

提交或发布前统一运行：

```bash
npm run verify
```

它会执行：

- `npm run check`：检查所有 JS/MJS 文件语法。
- `npm run smoke`：验证 narrative memory、handoff、异常察觉事件和导出。
- `npm run verify:version`：检查版本号一致性。

打包是单独的发布步骤：

```bash
npm run package:zip
```

## 版本来源

当前扩展版本必须在这些文件中保持一致：

- `package.json`
- `manifest.json`
- `data/version.js`

`scripts/verify-version.mjs` 会强制检查这一点。如果发布时只改了其中一个文件，`npm run verify` 会失败。

## 发布卫生

创建 release 前：

1. 更新 `package.json`、`manifest.json` 和 `data/version.js` 中的版本。
2. 更新 `CHANGELOG.md`。
3. 运行 `npm run verify`。
4. 运行 `npm run package:zip`，按需把 zip 附到 GitHub release。
5. 确认 README 和发布检查清单指向当前 alpha 版本。
6. 创建 tag，并在 GitHub 上创建 pre-release。

## 范围控制

Alpha 阶段保持改动小而可回退：

- 优先做文档、验证和兼容性修复，不急着做大型 UI 重构。
- 除非有明确用户流程需要，否则不引入后端、数据库、账号系统或构建流水线。
- SillyTavern 集成点要尽量在 Debug 中可观察，方便用户带着证据反馈兼容问题。
