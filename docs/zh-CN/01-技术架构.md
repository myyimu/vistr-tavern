# 技术架构

## 架构决策

第一阶段应该做 SillyTavern Extension。

不要一开始就做独立 Web App、NestJS 后端、用户系统或 PostgreSQL。只有当“真人入侵 AI 群像”的核心闭环被验证后，这些才有必要。

## 系统总览

```text
SillyTavern
  -> VistrTavern Extension
      -> UI Layer
      -> Intrusion Engine
      -> Narrative Engine
      -> Memory Layer
      -> Writer Adapter
      -> Storage Adapter
```

## 运行时职责

SillyTavern 负责：

- 现有聊天界面。
- 角色卡。
- 上下文组装。
- 模型调用。
- 常规 AI 角色扮演流程。

VistrTavern 负责：

- 某一时刻由谁控制角色。
- 控制者身份是隐藏还是揭示。
- 如何记录真人入侵事件。
- 如何把 AI 反应绑定到入侵上下文。
- 如何把一次会话整理成结构化叙事素材。

## MVP 模块结构

这个仓库本身就是扩展目录。安装到 SillyTavern 时，应放在 `extensions/vistr-tavern/`。

```text
vistr-tavern/
  manifest.json

  style.css

  core/
    intrusionEngine.js

  data/

  prompts/

```

## 核心数据流

```text
消息产生
  -> VistrTavern 观察消息
  -> Intrusion Engine 判断控制者
  -> Narrative Memory 记录控制者、场景、张力、可见性
  -> Narrative Engine 更新扰动状态
  -> Writer Adapter 导出结构化素材
```

## 主要组件

### UI Layer

提供：

- VistrTavern 面板。
- 角色接管控制。
- 倒计时显示。
- 场景和张力控制。
- 沉浸模式 / 上帝模式切换。
- 导出控制。

UI 不能成为产品中心。它只是用来操作入侵系统。

### Intrusion Engine

最核心的控制模块。

职责：

- 开始角色接管。
- 结束角色接管。
- 跟踪限时控制。
- 自动恢复 AI 控制权。
- 保存可见性模式。
- 发出入侵生命周期事件。

关键 API：

```js
startIntrusion({ characterId, durationMs, visibility, mode })
endIntrusion(characterId, reason)
getController(characterId)
isHumanControlled(characterId)
getActiveIntrusions()
tick(now)
```

### Narrative Engine

把会话解释为一次叙事扰动实验。

职责：

- 跟踪当前场景。
- 跟踪张力值。
- 识别扰动窗口。
- 标记 AI 对真人异常源的反应。
- 维护人物关系和世界状态变化。

MVP 阶段应该尽量保持手动或启发式分析。LLM 自动分析可以后加。

### Memory Layer

保存结构化证据，而不是只保存原始聊天文本。

职责：

- 消息记录。
- 入侵生命周期记录。
- 场景记录。
- 张力快照。
- 人物关系变化。
- 世界状态变化。
- 导出会话。

MVP 存储：

- 浏览器 localStorage 或 IndexedDB。
- 先设计 JSON schema。

后续存储：

- SQLite 用于本地档案。
- PostgreSQL 用于多人后端。

### Writer Adapter

把叙事记忆转换成创作者可用格式。

职责：

- Markdown 导出。
- JSON 导出。
- LLM Prompt 包导出。
- 场景卡。
- 关键事件列表。
- 人物关系变化总结。
- 世界状态变化总结。

MVP 阶段 Writer Adapter 不负责生成最终小说正文。它只为 GPT、Claude、DeepSeek 或人类作者准备素材。

### Storage Adapter

把持久化逻辑和核心逻辑分离。

MVP：

```text
Memory object -> storageAdapter -> localStorage / IndexedDB
```

后续：

```text
Memory object -> storageAdapter -> SQLite / API backend / PostgreSQL
```

## 模式设计

### Intrusion Mode

主模式。

- 真人作为异常源进入。
- 身份可以隐藏。
- 其他参与者不一定知道角色现在是 AI 还是真人。
- 系统记录叙事扰动。

### Director Mode

辅助模式。

- 真人主动修正或引导角色。
- 身份可以明确可见。
- 适合作者驱动的场景。

Director Mode 不能取代 Intrusion Mode 成为项目核心身份。

### Immersion Mode

- 游玩过程中不显示 AI / 真人标签。
- 场景结束后统一揭示控制历史。

### God Mode

- 明确显示 AI / 真人控制者标签。
- 用于调试、分析和写作整理。

## 未来后端架构

只在 Phase 3 或 Phase 4 之后引入。

```text
Client
  -> SillyTavern Extension / Web UI
  -> VistrTavern API Gateway
  -> Session Service
  -> Intrusion Orchestrator
  -> Narrative Memory Service
  -> LLM Provider Adapter
  -> PostgreSQL
  -> Object Storage
```

后端技术：

- Node.js。
- NestJS。
- Socket.IO。
- PostgreSQL。
- SQLite 只用于本地优先模式。

## MVP 不做什么

- 不做账号系统。
- 不做支付系统。
- 不做多人服务器。
- 不做公开角色市场。
- 不做完整 Web App。
- 不做自动小说生成。
