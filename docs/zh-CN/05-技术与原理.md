# 技术与原理

本文说明 VistrTavern 背后的工程机制和叙事假设。

## 工程技术

### SillyTavern 扩展运行时

VistrTavern 作为 SillyTavern 浏览器扩展运行。扩展通过 `manifest.json` 声明 JavaScript 入口、样式文件和生成拦截器。

当前入口包括：

- `index.js`：初始化运行时状态、UI、存储和 SillyTavern 事件绑定。
- `style.css`：控制浮动面板样式。
- `generate_interceptor`：注册 `VistrTavernPromptInterceptor`，用于生成前的连续性注入。

### 事件驱动的入侵状态

入侵引擎是事件驱动的。它不直接写叙事记忆，而是发出生命周期事件：

```text
intrusion:start
intrusion:end
```

这样可以把“控制权状态”与“记忆记录、导出、分析”解耦。

### 角色控制状态机

核心控制模型是一个小状态机：

```text
AI controlled
-> human controlled
-> AI recovered
```

每次 intrusion 会记录：

- 被接管角色
- 开始时间
- 预计结束时间
- 实际结束时间
- 结束原因
- 可见性
- 控制模式
- 恢复后的 awareness 模式

### 结构化叙事记忆

VistrTavern 不只保存原始聊天文本。每条关键消息都会绑定结构化上下文：

```text
message
-> session
-> scene
-> intrusion
-> handoff
-> controller
-> visibility
-> tension
```

这让后续分析、导出、prompt 注入和连续性处理有稳定依据。

### Continuity Handoff

当控制权从真人回到 AI 时，VistrTavern 会根据这次入侵窗口生成 continuity handoff。

handoff 会整理：

- 真人控制角色的行为
- 入侵期间的 AI 反应
- 场景上下文
- awareness 模式
- 下一次 AI 生成需要遵守的连续性规则

它的作用是避免模型把真人入侵当成临时噪声。

### 生成前上下文注入

VistrTavern 使用 SillyTavern 的 generation interceptor，在生成前插入待处理的 continuity handoff。

流程是：

```text
入侵结束
-> 生成 continuity handoff
-> 下一次生成开始
-> 待处理 handoff 被插入 chat context
-> 收到 AI 回复
-> handoff 标记为 consumed
```

被注入的消息会带上 VistrTavern 元数据，用来避免重复插入。

### Adapter 边界

代码尽量把 SillyTavern 集成限制在较窄的适配层：

- `StorageAdapter`：优先写入 chat metadata，失败时回退到 localStorage。
- `ExportWriter`：把叙事记忆转换成 Markdown 或 JSON。
- `VistrTavernPromptInterceptor`：处理 SillyTavern 生成前上下文注入。

这样 SillyTavern API 变化时，主要修改适配层，而不是重写叙事核心。

## 叙事和 AI 原理

### AI 群像稳定性

AI 角色扮演容易变得稳定、礼貌、连贯和可预测。模型倾向于维持上下文、平滑冲突，并沿着预期方向继续。

VistrTavern 的假设是：真人可以扰动这种稳定生成轨道。

### 真人作为异常源

真人控制的角色可能引入：

- 迟疑
- 矛盾
- 节奏变化
- 情绪风险
- 不完全合理但有意图的行为
- 动机模糊

这些行为可能激发纯 AI 群像不容易自然产生的反应。

### 扰动窗口

intrusion 是有时间边界的。VistrTavern 观察的是三个阶段之间的差异：

```text
入侵前
入侵中
AI 恢复后
```

产品价值来自记录一个短暂真人扰动如何改变后续剧情。

### 身份不确定性

匿名入侵保留了世界内的不确定性。其他角色不一定知道这个角色正在被真人控制。

这会让故事从世界内部解释异常行为：

```text
异常行为
-> 怀疑
-> 误读
-> 冲突
-> 关系或世界状态变化
```

### 叙事状态锚定

大语言模型在扰动后可能回到之前的稳定轨道。continuity handoff 的作用是重新锚定新状态：

```text
These events are canonical.
Do not ignore, reset, or rewrite them.
Continue from their emotional and world-state consequences.
```

这是从真人扰动过渡到 AI 继续剧情的关键桥梁。

### Awareness 模式

恢复后的 awareness 控制角色对异常控制事件的理解程度。

- `none`：角色在世界内不知道真人控制这件事。
- `subtle`：角色可以感到迟疑、记忆断片或失控感。
- `explicit`：角色可以把外部意志识别为世界内体验。

默认使用 `none`，用于保持沉浸感。

### 异常自我察觉

当 recovery awareness 不是 `none` 时，VistrTavern 会把这次恢复处理成一个可玩的“异常察觉”节点，而不只是普通连续性提示。

handoff 可以作用于：

- 恢复控制的角色
- 旁观 AI 角色
- 恢复角色和旁观者两者

插件不会伪造一条 AI 聊天消息。它会要求下一条相关模型回复包含一段简短斜体内心独白。

示例：

```text
*Why did I say that? That did not feel like me.*
*That sentence came from my mouth, but it did not feel born from my own will. Is this world truly stable?*
```

同时，这个节点会被记录为结构化扰动事件：

- `self_anomaly_awareness`
- `observer_anomaly_awareness`

## 版本和维护边界

VistrTavern 尽量使用 SillyTavern 文档化的扩展钩子，但仍然依赖 SillyTavern 的运行时行为。

这些部分在不同版本之间可能需要兼容性检查：

- generation interceptor 行为
- 事件名和事件 payload
- chat message 结构
- chat metadata 持久化

项目应该继续把这些依赖隔离在适配层，而不是让 SillyTavern 特定假设扩散到叙事引擎里。
